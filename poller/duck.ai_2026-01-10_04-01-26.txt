## 1. What the hardware actually delivers

A USB‑HID keyboard sends **reports** at a fixed interval (the “polling rate”).  
Typical values are 125 Hz, 250 Hz, 500 Hz or 1000 Hz.  
Each report contains **the complete set of keys that are currently pressed** (a bitmap of usage IDs) and, for the *primary* key, a repeat flag that the firmware sets after the driver‑defined repeat delay.

**What is *not* in the report**

| Missing information | Why it cannot be inferred from a single report |
|----------------------|-----------------------------------------------|
| The exact moment each key changed state | The report is a *snapshot*; it tells you only the *current* state, not the transition times. |
| The order in which multiple keys changed between two reports | If two keys go from up→down between report N‑1 and report N, the bitmap simply shows both as down; the order is lost. |
| How many times the repeat flag toggled during the interval | The flag is either set or not when the report arrives; rapid repeats that occur between two reports are collapsed into a single “repeat‑present” bit. |

Thus the raw data that Windows receives is already **lossy**. The OS cannot magically reconstruct a total order; it can only *guess* based on timing heuristics (e.g., “the first key that became down is the one that changed between the previous bitmap and the current one”). Those guesses are good enough for normal UI interaction, but they are **not** a faithful log of every transition.

---

## 2. What `GetAsyncKeyState` actually returns

```cpp
SHORT GetAsyncKeyState(int vKey);
```

* **Low‑order bit (0x0001)** – Current physical state (down = 1).  
* **High‑order bit (0x8000)** – “Transition since the last call on this thread”.

Important consequences:

1. **Per‑thread transition flag** – The flag is cleared *when your thread reads it*. If you poll every 1 µs, the flag will be set for **any** change that happened since the previous read, but it gives you **no count**. A rapid tap‑release‑tap sequence will still set the flag only once.

2. **No per‑key ordering** – Each virtual‑key code has its own flag. When you query `VK_A` and `VK_B` you get two independent snapshots. There is no global timestamp that tells you whether A changed before B or vice‑versa.

3. **Repeats are invisible** – While a key is held, the low‑order bit stays set. The repeat flag that the driver generates is **not** reflected in `GetAsyncKeyState`; you only see a continuous “down” state.

Because the API is built on top of the same lossy bitmap that the driver receives, it cannot magically restore the missing information.

---

## 3. Why a faster poller does **not** solve the problem

Assume the keyboard reports at 1 kHz (every 1 ms).  
Even if you poll at 1 MHz (every 1 µs), the data you read from `GetAsyncKeyState` can only change when the underlying bitmap changes – i.e., at most once per millisecond.

### Worst‑case scenario

| Event sequence (within one 1 ms report interval) | What you can observe |
|---------------------------------------------------|----------------------|
| `A` down → `A` up → `A` down (three transitions) | Transition flag for `A` is set **once**; you cannot know that three changes occurred. |
| `A` down, 0.3 ms later `B` down, 0.3 ms later `A` up | After the report arrives you see both `A` and `B` down. The order (A→B→A‑up) is lost. |
| Key held long enough for driver to generate 5 repeats | You see `A` continuously down; no repeat count is exposed. |

No amount of polling can create information that never existed in the source data. The only way to capture every transition is to **receive the raw HID reports** *before* they are collapsed into a bitmap, which is what a low‑level hook (`WH_KEYBOARD_LL`) or a raw‑input (`WM_INPUT`) handler does.

## 4. `WH_KEYBOARD_LL` vs. `GetAsyncKeyState`

| Aspect | `WH_KEYBOARD_LL` hook | `GetAsyncKeyState` |
|--------|----------------------|--------------------|
| Source | Receives **individual** low‑level events (scan‑code, key‑up/down, repeat flag) directly from the input stack. | Reads the **per‑thread aggregated state table** that the system builds *after* processing those events. |
| Ordering | Preserves the exact order of each event, with a timestamp supplied by the system. | No ordering information; only a binary “has changed since last read”. |
| Visibility across integrity levels | Delivered to any process that has the hook installed, regardless of the foreground window’s integrity (unless the system blocks hooks entirely). | Visible to any process because it queries the global state, but it cannot distinguish multiple rapid transitions. |
| Repeat handling | Repeat flag is part of the event stream, so you see each repeat as a separate `WM_KEYDOWN`. | No repeat information; you only see a steady “down” state. |

So the hook **does** use the same underlying hardware source, but it intercepts the data *before* Windows collapses it into the per‑thread bitmap that `GetAsyncKeyState` reads.

*is **not present** in the data that `GetAsyncKeyState` exposes.*
