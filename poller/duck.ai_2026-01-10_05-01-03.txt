Explanation — what is available vs what can be lost
- What GetAsyncKeyState provides:
  - Current up/down state for each virtual key at the moment of the query.
  - A transition bit per thread indicating whether the key changed since the last call on that thread (but that bit is per-thread and not reliable across focus/elevation boundaries for monitoring other processes/windows).
- What the hardware/stack provides:
  - USB HID keyboards present periodic snapshots (key matrix or array of pressed key codes) at the device’s polling interval (~125 Hz typical, high-end 1000 Hz for gaming keyboards). Each snapshot is the sampled instantaneous state.
  - Firmware on the keyboard (and HID driver) may implement debouncing, firmware-level n-key rollover behavior, or even event coalescing/report compression.
  - The USB/host stack can drop or merge very fast transitions that occur entirely between two samples.
- Consequently:
  - If you poll at or above the device’s polling frequency and your poll is aligned and low-jitter, you can observe the same sampled states the OS sees and thus infer most transitions and ordering that are representable by sampled snapshots.
  - If two keys change between two successive samples (A up then B down then A down) such that intermediate states never appear in any sample, ordering is ambiguous or lost.
  - Very short presses shorter than the inter-sample interval can be missed entirely.
  - Key repeats: since repeats are synthesized by Windows based on observed transitions and system repeat parameters, you can emulate repeats if you detect the initial make and keeping state; but if the initial make was missed, repeat inference fails. Timing jitter in polling and timestamp resolution affects accuracy.
  - Per-thread transition bits are unreliable for a global monitor; they’re intended for caller thread use.

Practical limitations and edge cases
- Sampling theorem: you can only recover transitions whose temporal frequency is below half the sampling frequency and whose intermediate states are visible at sample times. So with 1000 Hz polling, events briefer than ~0.5–1 ms or sequences faster than that can be missed.
- Firmware debouncing: keyboards may suppress quick toggles; you’ll only see debounced results.
- HID report format: many keyboards send lists of currently pressed keys (array of keycodes). If more than N simultaneous keys exist, overflow behavior may hide some keys.
- Keyboard hardware/driver may implement ghost suppression and NKRO in ways that change what snapshots look like compared to raw switch scans.
- USB Host/OS latencies: even if the device polls at 1000 Hz, OS/driver scheduling may introduce jitter; achieving true ≥1000 Hz polling from user-mode is challenging without real-time priority.
- Elevated/Integrity boundaries: GetAsyncKeyState’s per-thread transition bit is less useful across privileges, but the raw up/down state is available system-wide for the calling session; Windows does not block that the same way it can for hooks. That makes polling potentially effective across elevated windows in many cases.

Recommended reconstruction approach (prescriptive)
1. Polling:
   - Target at least the device poll rate (1000 Hz for gaming boards), use a high-resolution busy-wait loop with QueryPerformanceCounter or a real-time thread priority to minimize jitter.
2. Observe:
   - On each sample, read GetAsyncKeyState for keys of interest (or use GetAsyncKeyState for all VK codes 1..254).
   - Record timestamped bitmap of down keys.
3. Reconstruct transitions:
   - Compare consecutive bitmaps: bit 0→1 = make (down), 1→0 = break (up). When multiple bits change between samples, order is ambiguous — you must pick an order (e.g., timestamp midpoint) but it’s not authoritative.
4. Repeats:
   - If you detect a make and the key remains down across the system repeat delay, synthesize repeated key events at the system repeat rate.
5. Handle ambiguities:

6. Validation:

- Firmware coalescing/debouncing => you only see processed state (not raw switch events).
- HID overflow/NKRO limits => missing keys.
